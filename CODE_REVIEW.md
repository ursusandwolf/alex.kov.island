## 🔴 КРИТИЧЕСКИЕ АЛГОРИТМИЧЕСКИЕ УЗЛЫ (Priority P0)

### 1. Квадратичная сложность кормления ($O(N^2)$)
**Файл:** `src/main/java/com/island/content/FeedingService.java`

**Проблема:**
В каждой клетке для каждого хищника ($N_{predator}$) перебираются все животные ($N_{all}$) как потенциальные жертвы.
Сложность: $O(N_{predator} \times N_{all})$. При 1000 животных в клетке это 1,000,000 проверок за такт.

**Рекомендация:**
- Сгруппировать потенциальных жертв по типам (Map<SpeciesKey, List<Animal>>).
- Вместо полного перебора проверять только те виды, которые входят в рацион хищника.

---

### 2. CopyOnWriteArrayList в Cell
**Файл:** `src/main/java/com/island/model/Cell.java`

**Проблема:**
Использование `CopyOnWriteArrayList` для хранения животных. Каждое добавление (`addAnimal`) или удаление (`removeAnimal` / съедение) создает полную копию массива. Это $O(N)$ на каждую операцию изменения.
При интенсивном движении и поедании это "убивает" CPU и перегружает Garbage Collector.

**Рекомендация:**
- Заменить на `ArrayList` (так как доступ к клетке уже защищен `ReentrantLock`) или использовать `ConcurrentLinkedQueue`.

---

### 3. Сортировка в каждой клетке ($O(N \log N)$)
**Файл:** `src/main/java/com/island/content/FeedingService.java`

**Проблема:**
```java
animals.sort((a, b) -> Double.compare(b.getWeight(), a.getWeight()));
```
Выполняется в каждой клетке на каждом такте. При большой популяции сортировка тысяч элементов 2000 раз в секунду потребляет значительные ресурсы.

**Рекомендация:**
- Поддерживать список животных уже отсортированным или использовать `PriorityQueue`.
- Либо сортировать только при значительных изменениях состава.

## 🟠 ПРОБЛЕМЫ ЭФФЕКТИВНОСТИ (Priority P1)

### 4. Аллокация объектов в горячих циклах
**Файлы:** `FeedingService.java`, `ConsoleView.java`, `MovementService.java`

**Проблема:**
В каждом такте создаются тысячи временных объектов:
- `new ArrayList<>(cell.getAnimals())` — копия списка в каждой клетке.
- `stream().filter().collect()` — создание потоков и коллекторов.
- `biomassMap` в `ConsoleView`.

**Рекомендация:**
- Использовать переиспользуемые буферы или итераторы, не создающие мусор.
- Перейти от Stream API к обычным циклам `for-i` в критических местах.

---

### 5. Избыточность Locking при параллельной обработке Чанков
**Файл:** `Island.java`, `MovementService.java`

**Проблема:**
Система использует разбиение на Чанки для параллелизма, но при этом каждая операция в `Cell` все равно требует `lock.lock()`. Это может привести к значительным задержкам на синхронизации (Lock Contention).

**Рекомендация:**
- Перейти на Actor-model или использовать Double-Buffering (расчет состояния в одном буфере, применение — в другом), чтобы минимизировать блокировки.

## 🟡 ТЕХНИЧЕСКИЙ ДОЛГ (Priority P2)

### 6. Глобальный SimulationConstants
**Проблема:**
Многие параметры (BASE_METABOLISM_PERCENT) являются `public static`. Это затрудняет проведение параллельных тестов с разными настройками.

**Рекомендация:**
- Перенести константы в объект `Configuration`, который пробрасывается через конструкторы.
