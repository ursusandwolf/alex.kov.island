## 🚨 1. Анализ баланса экосистемы (CRITICAL)

### Проблема
Текущая симуляция нестабильна. Наблюдаются два сценария коллапса:
1.  **Быстрое вымирание хищников:** Хищники тратят больше энергии на движение, чем получают от редкой охоты.
2.  **Перенаселение и крах:** Травоядные размножаются быстрее, чем хищники могут их контролировать -> переедание растений -> массовый голод -> вымирание всех.

### Причины дисбаланса
Разработчик отошел от проверенных параметров в `FULL_TASK.md` в пользу экспериментальных значений без математического обоснования.

#### Сравнение параметров (ТЗ vs Реальность)

| Параметр | FULL_TASK.md (Оптимально) | Текущий код (dev) | Проблема |
|----------|---------------------------|-------------------|----------|
| **Wolf Energy Gain** | 40-50 units | 15-20 units | ❌ Хищник не насыщается одной жертвой |
| **Wolf Hunt Chance** | 30-40% | 85-95% | ❌ Слишком высокая эффективность, быстрое истребление |
| **Rabbit Reproduction** | Every 5-7 ticks | Every 2-3 ticks | ❌ Взрывной рост популяции |
| **Grass Growth Rate** | 10% per tick | 2% per tick | ❌ Растения не восстанавливаются |
| **Movement Cost** | 0.5 units | 2.0 units | ❌ Движение слишком "дорогое" |

### Рекомендации по балансу
1.  **Вернуть константы из ТЗ:** Использовать значения из `FULL_TASK.md` как базовую конфигурацию (`species.properties.baseline`).
2.  **Внедрить формулу баланса:**
    ```java
    // Энергия должна сходиться:
    // (EnergyFromPrey * HuntChance) > (MoveCost * AvgMovesPerTick + BaseMetabolism)
    ```
3.  **Динамическая регуляция:** Добавить множитель сложности охоты в зависимости от плотности популяции (чем больше жертв, тем легче охотиться, но не линейно).

## 🏗 2. Архитектурный обзор и SOLID

### S - Single Responsibility Principle

#### ❌ Нарушение: SimulationBootstrap
Класс берет на себя слишком много ответственности:
- Загрузка конфигов
- Создание матрицы взаимодействий
- Инициализация мира
- Регистрация задач в цикле

**Решение:** Выделить `ConfigLoader`, `WorldFactory`, `TaskRegistry`.

#### ❌ Нарушение: FeedingService (Дублирование)
Существуют два класса с одинаковым именем:
- `com.island.service.FeedingService`
- `com.island.content.FeedingService`

**Риск:** Конфликты импорта, непонятно какой класс используется в `GameLoop`.
**Действие:** Удалить `com.island.service.FeedingService`, оставить расширенную версию в `content`, переименовав пакет или переместив в `service.logic`.

---

### O - Open/Closed Principle

#### ❌ Нарушение: Жесткая логика питания
В коде присутствуют конструкции:
```java
if (predator.equals("wolf") && prey.equals("rabbit")) { ... }
else if (predator.equals("bear") && prey.equals("fish")) { ... }
```
При добавлении нового животного нужно лезть внутрь сервиса.

**Решение:** Паттерн **Strategy**.
```java
public interface HuntingStrategy {
    boolean canHunt(Animal predator, Animal prey);
    double calculateSuccessRate(Animal predator, Animal prey);
}
```
Конфигурировать стратегии через JSON/XML или аннотации.

---

### D - Dependency Inversion Principle

#### ❌ Нарушение: Прямая зависимость от реализаций
Сервисы создают экземпляры других классов внутри методов.
```java
// Плохо
Animal baby = new Rabbit(type);
```

**Решение:** Использовать `AnimalFactory` или DI контейнер для создания сущностей. Это позволит легко подменять существ на мок-объекты в тестах баланса.

## 🧵 4. Потокобезопасность и Конкурентность

### Race Condition при поедании
**Сценарий:** Два хищника в одной клетке пытаются съесть одну жертву.
1.  Хищник А проверяет: жертва жива.
2.  Хищник Б проверяет: жертва жива.
3.  Хищник А удаляет жертву.
4.  Хищник Б пытается удалить жертву (ошибка или логический баг: хищник Б получил энергию, но жертвы нет).

**Текущее решение:** Частичная защита через `removeAnimal()` возвращающий boolean.
**Проблема:** Нет атомарности проверки "жива ли" + "удалить".

**Рекомендация:**
Использовать явную синхронизацию на объекте ячейки для блока "охота":
```java
synchronized(cell.getLock()) {
    if (prey.isAlive() && cell.contains(prey)) {
        cell.removeAnimal(prey);
        predator.feed(prey);
    }
}
```
