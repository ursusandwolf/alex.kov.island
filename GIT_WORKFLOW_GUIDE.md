# Гайд по работе с ветками Git и слиянию изменений

## 📋 Текущая ситуация

Вы находитесь в текущей ветке (скорее всего `main` или `master`), где был выполнен рефакторинг. Теперь нужно понять, как правильно слить изменения на GitHub и продолжить разработку.

---

## 🌳 Логика работы с ветками (Git Flow упрощенный)

### 1. Основные типы веток

| Тип ветки | Назначение | Время жизни |
|-----------|------------|-------------|
| `main` / `master` | Основная стабильная версия | Постоянно |
| `feature/*` | Разработка новой функциональности | До завершения фичи |
| `bugfix/*` | Исправление ошибок | До исправления |
| `refactor/*` | Рефакторинг кода | До завершения |
| `release/*` | Подготовка к релизу | До релиза |

### 2. Рекомендуемый рабочий процесс

```
main (стабильная версия)
  │
  ├─── feature/refactoring (вы здесь сейчас)
  │       └── [ваш рефакторинг]
  │
  └─── feature/new-animation (будущая фича)
          └── [новая разработка]
```

## 🔄 Пошаговая инструкция: Слияние изменений на GitHub

### Шаг 1: Проверьте текущее состояние

```bash
# Посмотреть текущую ветку
git branch

# Посмотреть статус изменений
git status

# Убедиться, что все закоммичено
git log --oneline -5
```

### Шаг 2: Запушьте изменения на GitHub

```bash
# Если ветка еще не на удаленном сервере
git push -u origin <имя-вашей-ветки>

# Пример:
git push -u origin main
# или
git push -u origin refactor/improve-code
```

### Шаг 3: Создайте Pull Request (PR) на GitHub

1. Перейдите на страницу вашего репозитория на GitHub
2. Нажмите **"Compare & pull request"** (появится автоматически после пуша)
3. Заполните:
   - **Base**: `main` (или `master`) - куда сливаем
   - **Compare**: ваша ветка с изменениями
   - **Title**: "Рефакторинг кода: улучшение архитектуры"
   - **Description**: кратко опишите изменения (можно скопировать из REFACTORING_REPORT_UK.md)
4. Нажмите **"Create pull request"**

### Шаг 4: Ревью и мерж

**Вариант А: Вы работаете один**
- Просто нажмите **"Merge pull request"** → **"Confirm merge"**
- Удалите ветку после слияния (кнопка "Delete branch")

**Вариант Б: Есть команда**
- Дождитесь ревью от коллег
- Исправьте замечания (если есть)
- После аппрува сделайте мерж

### Шаг 5: Обновите локальную копию

```bash
# Переключитесь на main
git checkout main

# Скачайте последние изменения с GitHub
git pull origin main

# Удалите старую ветку (опционально)
git branch -d <имя-вашей-ветки>
```

## 💻 Продолжение самостоятельной разработки

### Сценарий 1: Разработка новой фичи

```bash
# 1. Убедитесь, что вы в main и всё актуально
git checkout main
git pull origin main

# 2. Создайте новую ветку для фичи
git checkout -b feature/new-island-map

# 3. Разрабатывайте, делайте коммиты
git add .
git commit -m "Добавлен новый тип ландшафта"

# 4. Запушьте и создайте PR (как выше)
git push -u origin feature/new-island-map
```

### Сценарий 2: Исправление бага

```bash
# 1. От основной ветки создайте bugfix
git checkout main
git pull origin main
git checkout -b bugfix/fix-animal-movement

# 2. Исправьте баг, закоммитьте
git add .
git commit -m "Исправлена ошибка движения животных"

# 3. Запушьте и создайте PR
git push -u origin bugfix/fix-animal-movement
```

### Сценарий 3: Продолжение рефакторинга

```bash
# Если нужно продолжить рефакторинг после мержа
git checkout main
git pull origin main
git checkout -b refactor/part2-database

# Работайте дальше...
```

## 🎯 Золотые правила работы с ветками

### ✅ Делайте так:

1. **Одна фича = одна ветка**
   - Не смешивайте разные задачи в одной ветке
   
2. **Частые коммиты**
   - Коммитьте каждые 30-60 минут или после завершения логического блока
   - Пишите понятные сообщения коммитов

3. **Короткие названия веток**
   - `feature/user-auth` ✅
   - `feature/add-new-user-authentication-system-with-database` ❌

4. **Регулярно обновляйте ветку от main**
   ```bash
   git checkout feature/my-feature
   git merge main
   # или
   git rebase main
   ```

5. **Пушьте каждый день**
   - Не держите код только локально больше суток

### ❌ Избегайте этого:

1. **Работа напрямую в main**
   - Никогда не коммитьте в main напрямую (кроме самых простых правок)

2. **Долгоживущие ветки**
   - Ветка не должна жить больше 1-2 недель
   - Большие задачи разбивайте на подзадачи

3. **Конфликты слияния**
   - Регулярно мерджите main в свою ветку
   - Решайте конфликты сразу

4. **Непонятные сообщения коммитов**
   - `"fix"` ❌ → `"fix: исправлено падение при нулевой энергии"` ✅

## 📝 Шаблон сообщения коммита

```
<тип>: <краткое описание>

<детальное описание (опционально)>

<ссылка на задачу (опционально)>
```

**Типы коммитов:**
- `feat` - новая функциональность
- `fix` - исправление бага
- `refactor` - рефакторинг без изменения поведения
- `docs` - обновление документации
- `style` - форматирование, отступы (без изменений кода)
- `test` - добавление тестов
- `chore` - сборка, зависимости, инструменты

**Примеры:**
```
feat: добавлен новый тип животного - медведь

fix: исправлена ошибка переполнения при большой популяции

refactor: вынесена логика движения в отдельный класс

docs: обновлена документация по конфигурации видов
```

## 🔧 Полезные команды Git

```bash
# Посмотреть все ветки (локальные и удаленные)
git branch -a

# Посмотреть историю коммитов в виде графа
git log --oneline --graph --all

# Отменить последний коммит (оставив изменения)
git reset --soft HEAD~1

# Отменить изменения в файле
git checkout -- <файл>

# Временно сохранить изменения
git stash
git stash pop  # вернуть позже

# Посмотреть различия
git diff
git diff main..feature/my-branch

# Удалить ветку локально
git branch -d <ветка>

# Удалить ветку на GitHub
git push origin --delete <ветка>
```

- [ ] Код компилируется без ошибок
- [ ] Все тесты проходят

- [ ] Ветка актуальна (синхронизирована с main)
- [ ] Создан Pull Request с описанием

## 🆘 Что делать если...

### ...возник конфликт слияния?

```bash
# Git скажет какие файлы в конфликте
git status

# Открой файлы, найди маркеры конфликта:
# <<<<<<< HEAD
# твои изменения
# =======
# изменения из main
# >>>>>>>

# Вручную отредактируй файл, оставив нужный код

# После разрешения всех конфликтов:
git add <файлы>
git commit -m "resolve: разрешен конфликт слияния"
```

### ...случайно закоммитил в main?

```bash
# Создай новую ветку от текущего состояния
git branch feature/my-recovery

# Вернись в main и отмени коммит
git checkout main
git reset --hard HEAD~1

# Продолжай работу в новой ветке
git checkout feature/my-recovery
```

### ...нужно срочно исправить баг в production?

```bash
# Создай хотфикс от main
git checkout main
git pull origin main
git checkout -b hotfix/critical-bug

# Исправь баг, закоммить, запуш
# Создай срочный PR с пометкой [HOTFIX]
```
