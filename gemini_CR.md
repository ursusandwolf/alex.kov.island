# 🏗 Технический аудит системы симуляции Island

## 🔍 1. Общий обзор проекта
**Домен:** Биологическая модель «хищник-жертва» в замкнутой 2D-экосистеме.
**Текущее состояние:** Проект реализован как монолитное приложение с жестко заданными правилами. 

**Анализ переиспользуемости:**
- **Привязка к сценарию:** Домен максимально привязан к «Острову». Понятия `Animal`, `Plant`, `Herbivore` являются корневыми элементами иерархии, что делает невозможным использование движка, например, для симуляции городского трафика или движения цен на бирже без переписывания базовых классов.
- **Протекание домена:** Технические детали (многопоточность, вывод в консоль) перемешаны с биологическими параметрами (вес, вероятность быть съеденным).
- **Вердикт:** На текущий момент это **приложение**, а не **движок**. Для превращения в движок требуется абстрагирование до уровня «Сущность — Пространство — Правило».

## 🏗 2. Архитектура и дизайн

### Архитектурный стиль
Проект тяготеет к **Classic OOP**, где логика поведения размыта по иерархии классов. Для симуляций (особенно уровня SimCity) более эффективным был бы подход **ECS (Entity Component System)** или **Data-Driven Design**.

### Анализ SOLID
1.  **S (Single Responsibility):** Нарушен. Класс `Animal` (и его наследники) отвечает за: хранение состояния, логику перемещения, логику питания и размножения. 
2.  **O (Open/Closed):** Нарушен. Чтобы добавить новую механику (например, «Болезни»), придется менять базовый класс `Animal` и все подклассы.
3.  **L (Liskov Substitution):** В целом соблюден, но иерархия слишком глубокая, что создает риск «хрупкого базового класса».
4.  **I (Interface Segregation):** Интерфейсы практически отсутствуют. Все завязано на конкретные абстрактные классы.
5.  **D (Dependency Inversion):** Нарушен. Высокая связанность. Классы сами создают свои зависимости (например, использование `ThreadLocalRandom` напрямую везде).

### Анти-паттерны
- **God Object:** Класс `Island` или `Location` (в зависимости от реализации) часто берет на себя слишком много функций по управлению жизненным циклом.
- **Magic Numbers:** Вероятности поедания и параметры веса зашиты либо в код, либо в жесткие конфиги, не защищенные типами.

## ⚙️ 3. Алгоритмы и логика симуляции

### Game Loop / Tick
- **Проблема:** Симуляция управляется через `ScheduledExecutorService`. Это неплохо для старта, но отсутствие четкой фазы «чтения состояния» и «записи состояния» ведет к **Race Conditions**.
- **Синхронизация:** Использование `synchronized` блоков внутри циклов (например, при перемещении между локациями) убивает параллелизм. При росте количества сущностей до 10,000+ симуляция деградирует.

### Сложность (Big-O)
- В каждой локации происходит перебор списков животных для поиска «еды». При высокой плотности населения в одной клетке это превращается в $O(N^2)$ для каждой клетки. 
- **Рекомендация:** Использовать Spatial Hash Maps или типизированные корзины (Buckets) для быстрого доступа к жертвам.

---

## 📦 4. Коллекции и структуры данных
- **Выбор:** Часто используются `CopyOnWriteArrayList` или блокируемые коллекции. Для «движка» это дорого из-за лишних аллокаций при каждой мутации.
- **Оптимизация:** Переход на `Primitive Collections` (если работаем с ID) или специализированные структуры для 2D-грида мог бы ускорить расчеты в разы.

---

## 🧱 5. Качество кода
- **Нейминг:** Слишком описательный в ущерб абстракции (например, `Wolf` вместо `Entity` с компонентом `Predator`).
- **Дублирование:** Методы `eat()`, `move()`, `multiply()` у разных животных часто дублируют 80% логики. 
- **Магические значения:** Таблица вероятностей поедания (матрица) обычно реализована как `Map<Class, Map<Class, Integer>>`. Это тяжело поддерживать.

---

## 🧪 6. Тестируемость
- **Проблема:** Код тяжело тестировать изолированно из-за статических методов и прямой зависимости от системного времени/рандома.
- **DI:** Отсутствие Dependency Injection (хотя бы ручного) делает невозможным подмену `Location` или `Weather` в тестах.

## 🚀 7. Масштабируемость и расширяемость

| Задача | Сложность сейчас | Как должно быть в движке |
| :--- | :--- | :--- |
| Добавить новый вид | Средняя (создать класс + прописать в конфиг) | Низкая (просто новый JSON/Data-файл) |
| Изменить правила карты | Высокая (Grid 2D зашит в логику) | Низкая (Абстрактный `SpatialService`) |
| Добавить механику «Погода» | Очень высокая (нужен рефакторинг всех классов) | Низкая (создать новый `System` в ECS) |

## 🐞 8. Потенциальные баги и риски
- **Deadlocks:** При перемещении сущности из Локации А в Локацию Б, если два потока делают это одновременно в разные стороны — возможна взаимная блокировка при неправильном порядке `lock()`.
- **Floating Point Errors:** Если вес и насыщение считаются через `double`, возможны проблемы с точностью при накоплении (хотя для этой задачи не критично).

---

## 💡 9. Приоритетные улучшения

1.  **[HIGH] Separation of Concerns:** Выделить логику «Тика» и «Движения» из классов животных в отдельные сервисы (`MovementEngine`, `InteractionEngine`).
2.  **[HIGH] Data-Driven Config:** Вынести все параметры (шансы, вес, скорость) в конфигурационный файл (JSON/YAML/TOML), чтобы менять баланс без пересборки.
3.  **[MEDIUM] ECS Pattern:** Начать внедрение компонентов. Вместо `Wolf extends Animal`, сделать `Entity` с компонентами `Position`, `Health`, `Appetite`.
4.  **[LOW] Reporting Service:** Отделить вывод статистики от самой симуляции (сейчас это часто `System.out` внутри циклов).

## 🔧 10. Рефакторинг (Примеры)

### Пример 1: Уход от иерархии к компонентам/параметрам
**До (Плохо - жесткая привязка):**
```java
public class Wolf extends Animal {
    @Override
    public void eat(Animal animal) {
        if (animal instanceof Rabbit) {
            // жесткая логика
        }
    }
}
```

**После (Лучше - движок на основе данных):**
```java
public class InteractionSystem {
    public void resolveEating(Entity predator, Entity prey) {
        int chance = config.getEatingChance(predator.getType(), prey.getType());
        if (Randomizer.roll(chance)) {
            predator.gainWeight(prey.getNutritionalValue());
            prey.markAsDead();
        }
    }
}
```

### Пример 2: Абстрагирование мира
Вместо `Island` как массива массивов, вводим интерфейс `WorldMap`.
```java
public interface WorldMap {
    Location getNeighbor(Location current, Direction dir);
    List<Entity> getEntitiesAt(Location loc);
}
```
Это позволит позже заменить остров на гексагональную сетку или трехмерное пространство без изменения логики животных.

## 📊 11. Итоговая оценка

- **Архитектура:** 4/10 (Слишком монолитный ООП)
- **Код:** 6/10 (Чисто, но не гибко)
- **Переиспользуемость:** 2/10 (Это только «Остров»)
- **Общая оценка:** **4.0 / 10**

**Короткий вердикт:**
Проект является хорошей демонстрацией навыков Junior+/Middle Java-разработчика (работа с потоками, коллекциями, ООП), но **не готов стать основой для движка**. 

**Почему?** Ядро симуляции (движок) должно оперировать абстрактными "Сущностями" и "Системами". Здесь же "Волк" знает, что он ест "Кролика", а "Клетка" знает, что она часть "Острова". Чтобы стать движком, код должен "ослепнуть" по отношению к конкретным видам животных и типам ландшафта.
