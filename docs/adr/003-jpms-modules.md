# ADR-003: Инкапсуляция через JPMS (Java Platform Module System)

**Дата:** 2026-05-11  
**Статус:** Accepted  

## Контекст
Проект является многомодульной библиотекой/платформой. Без жесткого контроля видимости разработчики плагинов могут начать использовать внутренние классы движка (`Default*`, `*Internal`), что приведет к:
1. **Хрупкости API:** Любое изменение внутренних классов будет ломать плагины.
2. **Сложности поддержки:** Невозможно гарантировать инварианты, если внутреннее состояние доступно извне.
3. **Запутанным зависимостям:** Модули начинают зависеть от деталей реализации друг друга.

## Решение
Использовать модульную систему Java (JPMS, проект Jigsaw):
- Каждый модуль имеет свой `module-info.java`.
- Экспортировать только пакеты с интерфейсами и стабильными API.
- Пакеты `.internal` НЕ экспортировать.
- Использовать `requires` для явного указания зависимостей.

## Последствия
### Положительные (✅)
- **Строгая инкапсуляция:** Внутренние классы физически недоступны для кода из других модулей даже через рефлексию (по умолчанию).
- **Чистый API:** Разработчик плагина видит только то, что ему разрешено использовать.
- **Безопасное обновление:** Можно полностью переписать пакет `.internal`, не опасаясь сломать клиентский код.

### Отрицательные (❌)
- **Сложность конфигурации:** Нужно аккуратно настраивать `module-info.java`, особенно при работе с библиотеками, не поддерживающими JPMS.
- **Ограничение тестов:** Модульные тесты иногда требуют специальных настроек (`--add-opens`) для доступа к полям через рефлексию (например, Mockito).

## Альтернативы
- **Соглашение об именовании:** (например, пакеты `internal`) Не гарантирует защиту, часто нарушается разработчиками.
- **OSGi:** Слишком сложная и тяжеловесная система для данного проекта.
