# ADR-002: Использование SoA (Struct of Arrays) для хранения критических данных

**Дата:** 2026-05-11  
**Статус:** Accepted  

## Контекст
Симуляция оперирует тысячами сущностей (животные, растения, здания). Классический подход "массив объектов" (AoS - Array of Structures) приводит к:
1. **Cache Misses:** При итерации по всем сущностям для обновления одного поля (например, энергии) процессор подгружает в кэш весь объект, хотя нужны только 4-8 байт.
2. **GC Pressure:** Огромное количество мелких объектов в куче нагружает сборщик мусора.
3. **Thread Safety:** Сложность эффективного конкурентного обновления полей в разных потоках.

## Решение
Для наиболее часто изменяемых и массовых данных использовать модель **Struct of Arrays (SoA)**:
- Вместо списка объектов хранить данные в примитивных массивах: `long[] energies`, `int[] ages`.
- Использовать `AtomicLongArray` и `AtomicIntegerArray` для обеспечения потокобезопасности без тяжелых блокировок там, где это критично.
- Доступ осуществляется по индексу, равному `entityId`.

## Последствия
### Положительные (✅)
- **Производительность:** Значительное снижение промахов кэша процессора при массовых операциях.
- **Масштабируемость:** Возможность эффективно обрабатывать на порядок больше сущностей.
- **Снижение GC:** Примитивные массивы практически не создают нагрузки на сборщик мусора.

### Отрицательные (❌)
- **Сложность кода:** Прямая манипуляция массивами менее интуитивна, чем вызов методов объекта.
- **Риск рассинхронизации:** Нужно гарантировать, что индексы во всех массивах соответствуют одной и той же сущности.

## Альтернативы
- **Object-Oriented Storage:** (AoS) Слишком медленно для симуляций высокого разрешения.
- **Off-heap Memory:** Рассматривалось, но отклонено как избыточно сложное для текущего этапа разработки на Java.
