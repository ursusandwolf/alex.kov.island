# Code review: `alex.kov.island` (branch `dev`, commit `dca2edf12093b1f2dba9d5a9d7ac532f9a69e3f8`)

Ревью в стиле тимлида: приоритет — архитектура, корректность алгоритмов, поддерживаемость и качество модели данных.

## Общий вывод

Проект уже содержит хорошую образовательную базу: есть разбиение на слои, отдельные сущности для организма/животного/растения, конфигурация видов вынесена отдельно, и видно стремление учесть многопоточность. Но текущая версия всё ещё находится в состоянии, где фундаментальные решения мешают масштабированию: слишком много логики сосредоточено в `Cell`, `FeedingService`, `PreyProvider` и `SpeciesConfig`, а также много сырых типов, `null`-контрактов и неявных инвариантов.

Главная архитектурная проблема: доменная модель пока не выражает естественные отношения между сущностями. Сейчас виды животных в основном живут в `List`, а затем каждый сценарий заново фильтрует эти списки. Это работает на маленьком объёме данных, но быстро превращается в O(n)-сканирование почти в каждом месте.

## Критические замечания

### 1) `Cell` хранит всё в двух плоских списках, из-за чего почти любая операция становится линейной

Сейчас в `Cell` есть `List<Animal> animals` и `List<Plant> plants`, а поиск/подсчёт по виду выполняется перебором списка. Это видно в `addAnimal`, `getAnimalsBySpecies`, `countAnimalsBySpecies`. Такой подход уже сейчас создаёт лишнюю стоимость, а при росте числа существ станет главным узким местом.

Что рекомендую:
- хранить животных не только в общем списке, но и в индексированной структуре, например:
  - `Map<String, List<Animal>> animalsBySpecies`
  - или `EnumMap<SpeciesGroup, List<Animal>>`, если типы будут нормализованы;
- если нужны быстрые подсчёты, держать отдельный `Map<String, Integer>` счётчиков;
- для частых операций кормления/поиска жертвы использовать заранее подготовленные очереди/бакеты, а не каждый раз пересобирать выборку.

Это особенно важно, если ты сам уже предполагаешь разделение хищников и травоядных, а также классификацию крупных и мелких существ.

### 2) `SpeciesConfig` делает слишком много и нарушает SRP

`SpeciesConfig` одновременно:
- читает `species.properties`,
- парсит конфигурацию видов,
- хранит параметры растений,
- содержит магический список species keys,
- вычисляет производные атрибуты вида.

Это типичный случай нарушения Single Responsibility Principle. Конфиг должен быть либо загрузчиком, либо хранилищем, либо фабрикой DTO, но не всем сразу.

Что рекомендую:
- выделить `SpeciesPropertiesLoader`;
- выделить `SpeciesCatalog` / `SpeciesRegistry` как immutable-слой;
- убрать хардкод `String[] speciesKeys` из кода и читать виды из файла или enum/registry;
- перенести параметры растений в отдельную конфигурацию `PlantConfig`.

### 3) Сырые типы (`Map`, `List`, `Set`, `Function`) размывают контракт и усложняют сопровождение

В проекте много обобщений без параметров: `Map prey`, `List predators`, `List masterPool`, `Set getRegisteredSpecies()`, `Iterable getPreyFor(...)`. Это снижает читаемость и делает ошибки типов скрытыми до runtime.

Что рекомендую:
- везде заменить на параметризованные типы;
- для доменных ключей завести отдельный value object или хотя бы `enum`;
- для `AnimalFactory` использовать `Map<String, Function<AnimalType, Animal>> registry`.

### 4) Много `null` как результата — это плохой API для фабрик и сервисов

`AnimalFactory.createAnimal(...)` возвращает `null`, если вид не найден. То же касается `createAnimalWithEnergy(...)`. Это создаёт каскадные проверки по всему коду и увеличивает шанс NPE.

Что рекомендую:
- возвращать `Optional<Animal>` там, где это уместно;
- либо бросать доменное исключение при неизвестном ключе;
- либо ввести `SpeciesKey` и сделать невозможным вызов с невалидным значением.

## Алгоритмы и производительность

### 5) `PreyProvider` строит «общий буфет» через перемешивание, но потом всё равно перебирает его линейно

Логика выбора жертвы сейчас выглядит как:
1. собрать `masterPool`,
2. перемешать список,
3. потом в `getPreyFor` снова пройти по нему и отфильтровать подходящие объекты.

Это даёт лишние проходы по данным и смешивает три задачи:
- подготовка доступных объектов,
- учёт скрывающихся/защищённых,
- выборку по вероятности охоты.

Что рекомендую:
- разделить на бакеты:
  - `predators`,
  - `herbivores`,
  - `plants`,
  - `smallBiomass` / `largeBiomass`;
- для каждой категории держать отдельный список или очередь;
- если нужен рандомизированный выбор, использовать `ThreadLocalRandom` и выбор по индексам, а не массовое `shuffle()`.

### 6) Есть явный перекос в пользу линейных «обходов по всем», хотя доменная модель просит индексирование

`FeedingService` сначала копирует `cell.getAnimals()`, потом сортирует, потом каждый predator заново получает `PreyProvider`, а затем снова обходится по пулу. Это много проходов и много промежуточных коллекций.

Что рекомендую:
- собирать в клетке отдельные структуры:
  - животные по ролям,
  - растения по видам,
  - потенциальную добычу по размеру;
- оптимизировать до сценария `pick candidates -> score -> resolve`;
- если точность симуляции важнее скорости, всё равно лучше иметь предрасчитанные списки и индексы, чем каждый тик пересобирать всё заново.

### 7) В коде есть логика «крупных» и «мелких» существ, но она сейчас зашита магическими числами

Например:
- `getDynamicMetabolismRate()` делит существа на `w < 1.0`, `w > 300.0`, иначе средние;
- `PreyProvider` использует разные квоты для крупных хищников;
- `FeedingService` тоже масштабирует затраты по весу и скорости.

Это правильная идея, но реализация хрупкая: магические пороги разбросаны по классам.

Что рекомендую:
- выделить `SizeClass` (`SMALL`, `MEDIUM`, `LARGE`, `HUGE`);
- вычислять класс один раз в `AnimalType`;
- использовать этот класс везде: метаболизм, скорость, шанс охоты, порядок хода, приоритет поиска добычи.

## Качество кода

### 8) `Organism` содержит слишком много mutable state для базовой сущности

`currentEnergy`, `age`, `isAlive`, `isHiding` живут в одном абстрактном классе, но часть этого состояния не принадлежит всем организмам одинаково. Например, `isHiding` — это явно поведение/состояние животного, а не растения.

Что рекомендую:
- вынести скрытие в интерфейс/компонент только для животных;
- отделить общую физиологию (`energy`, `age`, `alive`) от поведения (`hiding`, `hibernating`, `canMove`);
- сделать `Organism` максимально тонкой базой.

### 9) Есть подозрение на неудачную инициализацию и контракт для `AnimalFactory`

Метод `createAnimalWithEnergy` сначала создаёт животное с максимальной энергией, а потом перезаписывает энергию. Это лишний шаг, а смысл метода двусмысленный: он создаёт животное или пересоздаёт его состояние?

Что рекомендую:
- сделать единый конструктор/фабричный метод, принимающий `energyFactor` или `currentEnergy`;
- убрать «сначала максимум, потом перезапись»;
- сделать контракт фабрики однообразным для всех сценариев: обычный взрослый, детёныш, экземпляр с заданной энергией.

### 10) `AnimalFactory` — это реестр, но оформлен как ручная карта с повторяющимися строками

Сейчас почти всё строится на `registry.put("wolf", Wolf::new)` и т.п. Это рабоче, но тяжело расширяется и легко рассинхронизируется с конфигурацией.

Что рекомендую:
- использовать один источник правды для ключа вида;
- либо `enum SpeciesKey`,
- либо `ServiceLoader`/registry initializer,
- либо статическую регистрацию в самом классе вида;
- добавить валидацию, что все виды из `SpeciesConfig` реально зарегистрированы в `AnimalFactory`.

## SOLID

### 11) SRP нарушается в нескольких местах

Основные нарушители:
- `SpeciesConfig`,
- `FeedingService`,
- `PreyProvider`,
- `Cell`.

Каждый из них знает слишком много о соседних слоях, а это уже бьёт по тестируемости и изменениям.

### 12) OCP соблюдён частично, но не до конца

Добавление нового вида сейчас требует:
- правок в `SpeciesConfig`,
- правок в `AnimalFactory`,
- вероятно, правок в `species.properties`,
- иногда правок в логике кормления/приоритета.

То есть система не закрыта к изменениям. Нужен реестр, который уменьшит число точек модификации.

### 13) DIP нарушается на уровне сервисов

`FeedingService` напрямую зависит от:
- `Island`,
- `InteractionMatrix`,
- `Cell`,
- конкретных классов `Caterpillar`, `Plant`, `Cabbage`, `Grass`.

Это очень сильная связность с конкретной реализацией. Лучше зависеть от абстракций:
- `PreyResolver`,
- `FoodSource`,
- `OrganismView`,
- `SpeciesRepository`.

## Поддерживаемость

### 14) Магические строки и магические числа распространены слишком широко

Примеры:
- `"wolf"`, `"rabbit"`, `"caterpillar"` и т.д.;
- `0.5`, `0.3`, `1.0`, `300.0`;
- доли и коэффициенты в `FeedingService` и `Organism`.

Что рекомендую:
- вынести строки видов в enum или централизованный registry;
- вынести коэффициенты в конфиг или константы с говорящими именами;
- для всех порогов и бонусов оставить комментарий с объяснением доменной причины.

### 15) Логика симуляции пока разнесена без чёткой границы ответственности

Сейчас бизнес-правила частично в сущностях, частично в сервисах, частично в моделях. Это затруднит:
- отладку,
- написание тестов,
- добавление новых механик,
- балансировку симуляции.

Что рекомендую:
- сделать отдельный слой доменных правил: `InteractionResolver`, `MovementResolver`, `FeedingResolver`, `ReproductionResolver`;
- сущности оставить как данные + простые инварианты;
- сервисы сделать оркестраторами, а не хранилищами логики.

### 16) Тестов мало для такого объёма правил

Судя по дереву, есть только один интеграционный тест `StabilityIntegrationTest`. Для проекта, где постоянно меняются коэффициенты, правила поедания, репродукции и многопоточность, этого недостаточно.

Что рекомендую:
- unit-тесты на `AnimalType`, `AnimalFactory`, `SpeciesConfig`;
- тесты на `Cell` с конкурентным доступом;
- тесты на подбор добычи;
- тесты на правила репродукции;
- отдельные регрессионные тесты на балансные сценарии.

## Конкретные архитектурные рекомендации

### Разделить хищников и травоядных
Да, это хороший шаг. Лучше не только как подклассы, а и как отдельные коллекции в клетке или как отдельный индекс:

- `List<Animal> predators`
- `List<Animal> herbivores`

Так проще:
- считать приоритеты,
- собирать пул добычи,
- оптимизировать кормление,
- строить статистику.

### Добавить классификацию крупных и мелких существ
Лучше ввести `SizeClass`, чем держать пороги `1.0` и `300.0` в разных классах. Это уменьшит разъезд логики и даст единую точку настройки.

### Подумать о других коллекциях и структурах
Для симуляции лучше подойдут:
- `EnumMap` для плотных категорий,
- `Map<SpeciesKey, List<Animal>>` для индексации,
- `Deque` для очередей обработки,
- возможно, отдельный `BitSet`/матрица для быстрых разрешений взаимодействий, если видов немного и набор статичен.

### Упростить контракт фабрик и сервисов
Вместо `null` — явный результат. Вместо повторного сканирования — индексы. Вместо «универсальных списков» — структурированные группы.

## Приоритет внедрения

1. Убрать сырые типы и `null`-контракты.
2. Перестроить `Cell` на индексированные коллекции по виду/роли.
3. Разделить `SpeciesConfig` на загрузчик и immutable-регистр.
4. Вынести `SizeClass` и унифицировать магические пороги.
5. Добавить тесты на ключевые правила и конкурентный доступ.
6. Сократить прямые зависимости `FeedingService` от конкретных классов растений и животных.

## Итог

Сейчас код уже демонстрирует направление, но архитектура пока «перегружена эволюцией»: каждое новое правило добавляется поверх предыдущего слоя без достаточной нормализации модели. Самая выгодная инвестиция здесь — не добавлять ещё больше логики в существующие классы, а сначала переупаковать данные: отдельные индексы, отдельные группы, отдельные конфиги и единый источник правды для видов и размеров.
